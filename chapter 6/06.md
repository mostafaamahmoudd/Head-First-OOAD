# You solve _big problems_ the same way you solve _small problems_

These steps apply to systems with 1000+ classes just as much as they do to a couple of classes.

1. Make sure your software does what the customer wants it to do.
2. Apply basic OO principles to add flexibility.
3. Strive for a maintainable, reusable design.

---

## It’s all in how you _look at_ the big problem

The best way to look at a big problem is to **see it as a collection of individual pieces of functionality**.

You can treat each of those pieces as an **individual problem** and apply what you already know.

Once you get one part of the application working, you can move on to another piece of functionality. At each step, you're applying the same basic principles.

You can solve a big problem by breaking it into many functional pieces and then working on each piece individually.

A big problem is really just a **collection of smaller problems**, where each piece of functionality is a problem on its own.

---

## The things you already know...

### Encapsulation

By encapsulating what varies, you make your application more flexible and easier to change.

Encapsulation helps with big problems too. The more you encapsulate, the easier it becomes to break a large application into independent pieces.

---

### Coding to an interface

Coding to an interface rather than an implementation makes software easier to extend.

In large applications, this reduces dependencies between components and keeps the system **loosely coupled**, which is always desirable.

---

### Requirements and understanding the system

The best way to get good requirements is to understand what the system is supposed to do.

If you understand each small piece, it becomes easier to combine them into a larger system that behaves correctly.

---

### Analysis in real-world context

Analysis ensures your system works in a real-world context.

With large systems, you usually start by analyzing individual pieces of functionality, then analyzing how those pieces interact.

---

### Maintainability

Great software is easy to change and extend, and fulfills customer needs.

Higher **cohesion** means each piece is more independent, making the system easier to manage.

---

## Understanding the system

### What is the system like?

One way to understand a system is to compare it to things you already know.

This is called **commonality** — what is similar?

---

### What is the system not like?

Another way is to identify what the system is _not like_.

This is called **variability** — what is different?

---

## What is a feature?

A feature is a **high-level description of what a system needs to do**. Features usually come from the customer.

A single feature can produce multiple requirements needed to implement it. So features help you understand the system at a high level.

- Feature → comes from the customer
- Requirement → implemented by the developer

---

### Feature vs Requirement

There is no strict universal difference. It depends on context:

- Some people: feature = big functionality, requirement = small implementation detail
- Others: feature and requirement are interchangeable

A common interpretation:

> A feature is satisfied by multiple requirements.

---

## Use cases and big picture thinking

Use cases can become too detailed and may cause you to lose sight of the big picture.

When designing systems, it’s important to **defer details as long as possible**.

---

## Use case diagrams

Use case diagrams help you understand the system at a high level without diving into details.

They act like **blueprints of the system**.

- Use features to ensure completeness
- Use diagrams to verify system coverage

Once both are complete, you can confidently start building.

---

### Actor definition

An **actor** is any external entity that interacts with the system (not necessarily a person).

Example: a user or even another system like a bank.

---

### System boundary

The box in a use case diagram represents the system boundary:

- Inside the box → you implement
- Outside the box → external actors

---

### Relationships (`<<include>>`, `<<extend>>`)

These define relationships between use cases, but they should not distract from the overall design.

The goal is still the **big picture**, not over-detailing relationships.

---

## Domain analysis

Domain analysis is describing a system using terms the customer understands.

It involves:

- identifying
- collecting
- organizing
- representing

information about the domain based on experts, existing systems, and real-world knowledge.

It helps ensure you're building only what belongs in the system.

---

## What is a design pattern?

A design pattern is a reusable solution to a common design problem.

Once you understand patterns, you can apply them to improve your system design and avoid messy, inflexible code.

Design patterns are usually applied **after breaking the system into smaller parts**.

---

## Solving big problems

1. Listen to the customer and understand what they want.
2. Create a feature list in customer language.
3. Validate features with the customer.
4. Build use case diagrams (system blueprint).
5. Break the system into smaller parts.
6. Apply design patterns where needed.
7. Use OO principles to implement each part.

---

## Bullet points

- Big problems are collections of smaller problems.
- Start with features and requirements.
- Features are high-level; requirements are detailed implementations.
- Commonality and variability help you analyze systems.
- Use case diagrams show the big picture.
- Domain analysis keeps you aligned with the customer language.
- Actors are external entities that interact with the system.
