# Good Design = Flexible Software

## Nothing Ever Stays the Same

### Change Is Inevitable

No matter how much you like your software right now, it’s probably going to change tomorrow.

The harder you make it for your software to change, the harder it becomes to respond to your customer’s changing needs.

---

## Abstract Classes

### Abstract Classes Are Placeholders for Real Implementations

Abstract classes define shared behavior, while subclasses provide the actual implementation.

An abstract class is essentially a **generic base class** that stands in for concrete implementation classes.

### Key Idea

- The **abstract class defines behavior**
- The **subclasses implement that behavior**

This allows you to:

- Reuse common logic
- Avoid duplicate code
- Extend behavior cleanly

---

## UML: Abstract Classes

In UML, a class is abstract when its **class name is italicized**.

### Example

```text
Instrument

serialNumber: String
price: double

getSerialNumber(): String
getPrice(): double
setPrice(float)
getSpec(): InstrumentSpec
```

When the class name appears in _italics_, it means the class is abstract.

---

## Reuse Through Abstraction

Whenever you find **common behavior** in two or more places, extract that behavior into an abstract class and reuse it through subclasses.

This reduces duplication and improves maintainability.

---

## UML Relationships

### Aggregation

A line with a diamond:

```text
---<>
```

represents **aggregation**.

Aggregation is a special form of **association**, meaning:

> One object is made up, in part, of another object.

---

### Generalization

A line with a hollow arrow:

```text
--->
```

represents **generalization**.

Generalization means:

- One class extends another
- The subclass inherits behavior from a more general class

This is UML’s way of showing **inheritance**.

---

## UML Reminder

- _Italic class name_ = abstract class
- Diamond = aggregation
- Hollow arrow = inheritance / generalization

---

## Questions About UML

**Q: Are there many more UML symbols to learn?**  
**A:** Yes, UML includes many more symbols and notations.

But you do **not** need to memorize everything.

Most developers are productive using:

- Classes
- Associations

The goal is **communication**, not memorization.

## Great Software Is Tested by Change

One of the best ways to judge whether software is well-designed is to **try changing it**.

If software is hard to change, the design likely needs improvement.

# OO Catastrophe

## What Is an Interface?

An **interface** has a dual role:

1. It defines behavior shared by multiple types
2. It becomes the preferred type to code against

---

## UML: Interfaces

In UML, an interface is represented as:

```text
<<interface>>
Athlete
```

The interface name is often shown in _italics_.

---

## Code to the Interface, Not the Implementation

Whenever your code interacts with related classes, you have two choices:

- Code directly to a concrete class (`FootballPlayer`)
- Code to the interface (`Athlete`)

The best practice is:

> **Code to an interface, not an implementation.**

---

## Why This Matters

Coding to an interface gives your application flexibility.

Instead of supporting only one class, your code supports:

- Existing subclasses
- Future subclasses
- New implementations not yet created

This makes software far easier to extend.

### Example

If your code works with `Athlete`, it automatically works with:

- `FootballPlayer`
- `BaseballPlayer`
- `TennisPlayer`
- Future types like `CricketPlayer`

---

## Encapsulation

Encapsulation has prevented more maintenance problems than almost any other OO principle.

Its power comes from **localizing change**.

Instead of letting changing behavior spread across the system, move it into one isolated place.

---

## Encapsulate What Varies

Whenever behavior is likely to change:

- Move it away from stable code
- Isolate it
- Protect the rest of the system

This is the principle:

> **Encapsulate what varies.**

This is much bigger than avoiding copy-paste.  
It protects your classes from unnecessary reasons to change.

---

## Single Responsibility Principle (One Reason to Change)

Each class should have **only one reason to change**.

This is one of the strongest indicators of clean design.

---

## Why This Matters

Poorly designed software breaks when requirements change.

Flexible software survives change because:

- Responsibilities are separated
- Classes are focused
- Each class handles one concern

---

## Spotting Design Problems

When a class has **multiple reasons to change**, it is usually trying to do too much.

That is your signal to:

- Split responsibilities
- Extract classes
- Delegate behavior
- Reduce coupling

Each resulting class should do **one thing well**.

---

## Example Insight: `serve()`

Your explanation here is strong.

If multiple classes repeat similar `serve()` logic:

- `Dessert`
- `IceCream`
- `Topping`

Then the behavior should be encapsulated into something like:

```text
DessertService
```

Now every class delegates serving to:

```text
DessertService.serve()
```

This means if serving changes, you only update one place.

That is:

- Encapsulation
- Reuse
- Single responsibility
- Reduced duplication

---

## Key Insight

Most of the time, **abstracting common behavior naturally leads to encapsulation**.

Abstraction helps identify commonality.  
Encapsulation protects variation.

Together, they produce flexible software.

---

# Core OO Principles from This Chapter

- **Encapsulate what varies**
- **Code to an interface rather than an implementation**
- **Each class should have only one reason to change**
